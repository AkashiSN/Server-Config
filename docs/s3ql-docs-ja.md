## S3QL について

S3QL は、[Google Storage](https://cloud.google.com/storage/docs)、[Amazon S3](https://aws.amazon.com/s3/)、[OpenStack](https://www.openstack.org/software/) などのストレージサービスを用いて、全データをオンライン上に保存するファイルシステムです。インターネットに接続できるコンピュータであればどこからでもアクセス可能な、動的で無制限の容量を持つ仮想ドライブを提供します。

S3QL はフル機能を備えた UNIX ファイルシステムであり、概念的には ext4 のようなローカルファイルシステムと区別がつきません。さらに、圧縮、暗号化、データ重複排除、イミュータブル(変更不可)ツリー、スナップショットといった追加機能を備えているため、オンラインバックアップやアーカイブ用途にとくに適しています。

S3QL の設計思想は「性能や機能の詰め込みよりも、シンプルさと優雅さを重視する」ことにあります。ソースコードは可読性と保守性を最大化するよう配慮されており、堅牢なエラー検出およびエラー処理が初期段階から組み込まれています。また、すべてのコンポーネントについて幅広い自動テストケースが付属しています。

### 特徴

- **透過性**: 概念的にはローカルファイルシステムと区別がつきません。ハードリンク、シンボリックリンク、標準的な UNIX パーミッション、拡張属性、最大 2 TB までのファイルサイズに対応しています。
- **動的サイズ**: S3QL ファイルシステムのサイズは必要に応じて動的に増減します。
- **圧縮**: データはストレージへの保存前に LZMA、bzip2、または deflate (gzip) アルゴリズムで圧縮可能です。
- **暗号化**: 圧縮後(アップロード前)に、256 ビット鍵の AES で暗号化できます。さらに SHA256 HMAC チェックサムによって改ざんから保護されます。
- **データ重複排除**: 複数のファイルが同一の内容を持つ場合、重複データは一度だけ保存されます。これはファイルシステム内のすべてのファイルに跨って機能し、ファイルの一部だけが一致する場合にも有効です。
- **イミュータブル(変更不可)ツリー**: ディレクトリツリーを「これ以上一切変更できない」状態にできます。バックアップ作成後に内容を改変されないよう保証するのに有用です。
- **コピーオンライト・スナップショット**: ディレクトリツリー全体を追加のストレージ容量を使わずに複製できます。どちらかのコピーが変更された場合のみ、その変更部分のみが追加のストレージ容量を消費します。最小限の容量で、複数時点のディレクトリ状態を保持するインテリジェントなスナップショットに使えます。
- **ネットワーク遅延に依存しない性能**: ファイル内容の読み書きを伴わない操作(ディレクトリ作成、ファイル/ディレクトリの移動・リネーム・パーミッション変更など)は、ネットワーク通信を伴わないため非常に高速です。これは S3QL がファイル/ディレクトリ構造全体をデータベースに保持し、ローカルにキャッシュしつつリモートコピーを非同期更新することで実現されます。
- **低帯域回線への対応**: ファイル内容を小さなブロックに分割し、ブロックをローカルキャッシュします。これにより読み書きに必要なネットワーク通信回数とデータ転送量を最小化します。

### 開発状況

S3QL は安定しており、本番環境での使用に適していると見なされています。バージョン 2.17.1 以降、S3QL はセマンティック・バージョニングを採用しています。後方非互換な変更(ファイルシステムのリビジョンアップグレードを伴うものなど)は、メジャーバージョン番号の増加で示されます。

### サポートされているプラットフォーム

S3QL は Linux 上で開発・テストされています。OS X、FreeBSD、NetBSD での稼動報告もあり、互換性維持を目指していますが、リリース前テスト環境が不足しているため非 Linux 環境での全バージョン動作保証はありません。不具合を見つけた場合は報告をお願いします。

### 貢献

S3QL のソースコードは [GitHub](https://github.com/s3ql/main) で公開されています。

## インストール

### 依存関係

- カーネル: Linux 3.9 以降
- [psmisc](http://psmisc.sf.net/) ユーティリティ
- [SQLite](http://www.sqlite.org/) 3.7.0 以降の開発ヘッダ
- [Python](http://www.python.org/) 3.10 以降(開発ヘッダ付きでインストール済みのもの)
- 追加の Python 依存は `pyproject.toml` に記載されており、通常は Python パッケージインストーラが自動解決します。

### ソースリリースからのインストール

リリース tarball を [GitHub](https://github.com/s3ql/s3ql/releases) からダウンロードし、[signify](https://github.com/aperezdc/signify) で検証します:

```
signify -V -m s3ql-XX.tar.gz -p s3ql-XX.pub
```

`s3ql-XX.pub`(署名鍵)は信頼できる情報源から入手する必要があります。各リリースには次のリリース用の署名鍵が `signify` ディレクトリに同梱されているため、最初に S3QL をインストールする際に一度だけ手動で入手すれば十分です。

tarball を検証したら展開し、生成された `s3ql-X.Y.Z` ディレクトリに移動します。あとは `pyproject.toml` を扱える任意の Python ツールでインストールできます。どれを使えばよいかわからない場合、[pipx](https://pipx.pypa.io/latest/) がおすすめです:

```
pipx install .
```

これで必要な Python 依存がダウンロードされ、専用の仮想環境に S3QL がインストールされます。

わずかな性能向上のために、S3QL コマンドを実行する前に `PYTHONOPTIMIZE` 環境変数を設定することを検討してください(ラッパースクリプトを使うなど)。これによりデバッグアサーションのチェックが無効化されます。残念ながら Python アプリケーションでこれを自動的に行う仕組みは存在しないようです(あれば S3QL がデフォルトで設定していたはずです)。

### Git からのインストール / S3QL の開発

S3QL リポジトリをクローンし、`developer_notes/setup.md` を参照してください。

## ストレージバックエンド

S3QL は、様々なサービスプロバイダやプロトコルでデータを保存するために複数の *バックエンド* をサポートしています。*ストレージ URL* はバックエンドとバックエンド固有の情報を指定し、S3QL ファイルシステムを一意に識別するものです。構文はバックエンドごとに異なり、以下で個別に説明します。

また、ストレージ URL を受け付ける S3QL コマンドは `--backend-options` パラメータも受け付け、バックエンド固有のオプションを渡すことができます。指定可能なオプションは各バックエンドの項目で説明されています。

すべてのバックエンドは `http_proxy`(平文 HTTP 用)および `https_proxy`(SSL 用)環境変数を尊重します。

> **注記**: バックエンドは互換性を保証しません。たとえばローカルバックエンドで保存したデータを他のツールで Amazon S3 にコピーして S3QL の S3 バックエンドでアクセスするといった使い方はできません。

### Google Storage

[Google Storage](https://cloud.google.com/storage/) は Google が提供するオンラインストレージサービスです。S3QL で利用するには [GCP Console API Library](https://console.cloud.google.com/apis/library/) で JSON API を有効化してください。

Google Storage バックエンドは OAuth2 認証または [ADC](https://cloud.google.com/docs/authentication/production)(Application Default Credentials)を使用します。

OAuth2 認証を使うには、バックエンドログインに `oauth2` を指定し、バックエンドパスワードに有効な OAuth2 リフレッシュトークンを指定します。リフレッシュトークンは [s3ql_oauth_client](http://www.rath.org/s3ql-docs/man/oauth_client.html#oauth-client) プログラムで取得できます。このプログラムが指示するブラウザ URL を開き、コードを入力して Google アカウントで認証すれば、リフレッシュトークンが表示されます。これは一度行えば良く、明示的に失効させない限り有効です。

ADC を使うには、バックエンドログインに `adc` を指定し、パスワードは任意の値で構いません。

Google Storage バケットは [Google Storage Manager](https://console.cloud.google.com/storage/browser) で作成できます。ストレージ URL は:

```
gs://<bucketname>/<prefix>
```

*bucketname* はバケット名、*prefix* は S3QL が使うすべてのオブジェクト名に付加される任意のプレフィックスです。これにより 1 つのバケット内に複数の S3QL ファイルシステムを格納できます。

Google Storage バックエンドは次のオプションを受け付けます:

- `ssl-ca-path=<path>`: システムのデフォルト証明書ストアの代わりに、指定した CA 証明書でサーバ証明書を検証します。ファイル(複数証明書含む)またはディレクトリ(1 ファイル 1 証明書)を指定できます。
- `tcp-timeout`: TCP 接続のタイムアウト秒数(デフォルト: 20 秒)。この時間リモートサーバとデータ交換がなければ TCP 接続を閉じて再確立します。

### Amazon S3

[Amazon S3](http://aws.amazon.com/s3) は [AWS](http://aws.amazon.com/) のオンラインストレージサービスです。バケットは [AWS Management Console](https://console.aws.amazon.com/s3/home) で作成する必要があります。最高のパフォーマンスを得るには、地理的に最も近いリージョンでバケットを作成することを推奨します。

ストレージ URL は次の形式です:

```
s3://<region>/<bucket>/<prefix>
```

*prefix* は任意のプレフィックスで、1 つのバケットに複数の S3QL ファイルシステムを格納できます。例:

```
s3://ap-south-1/foomart.net/data/s3ql_backup/
```

は *ap-south-1* リージョンの *foomart.net* バケットを指し、S3QL が保存するオブジェクト名はすべて *data/s3ql_backup/* というプレフィックスを持ちます。

バックエンドのログイン/パスワードは Amazon ウェブサイトのアカウントではなく、[My Account/Access Identifiers](https://aws-portal.amazon.com/gp/aws/developer/account/index.html) に表示される *AWS access key id* と *AWS secret access key* です。

Amazon S3 バックエンドは次のオプションを受け付けます:

- `no-ssl`: 暗号化(https)接続を無効にし、平文 HTTP を使用します。
- `ssl-ca-path=<path>`: 指定 CA でサーバ証明書を検証(上記同様)。
- `tcp-timeout`: TCP タイムアウト(デフォルト: 20 秒)。
- `sse`: サーバサイド暗号化を有効化。S3 サーバサイド暗号化のコストと利点はどちらも小さい可能性が高く、S3QL 自身のクライアントサイド暗号化には影響しません。
- `it`: 新しいオブジェクトに INTELLIGENT_TIERING ストレージクラスを使用。
- `ia`: 新しいオブジェクトに STANDARD_IA(低頻度アクセス)ストレージクラスを使用。
- `oia`: 新しいオブジェクトに ONEZONE_IA ストレージクラスを使用。
- `rrs`: 新しいオブジェクトに低冗長化ストレージを有効化(*ia* オプションを上書き)。このオプションを有効にする場合は、バックエンドでオブジェクトが失われるとデータ重複排除により後で保存されたデータまで失われる可能性があるため、**s3ql_verify** を定期実行することを強く推奨します(詳細は「データ損失を防ぐための重要なルール」参照)。

### OpenStack / Swift

[OpenStack](http://www.openstack.org/) はオープンソースのクラウドサーバスイートで、[Swift](http://openstack.org/projects/storage/) がそのクラウドストレージモジュールです。Swift/OpenStack ストレージは多くの企業が提供しています。

OpenStack バックエンドには認証 API が異なる 2 種類のストレージ URL があります。レガシー(v1)認証では:

```
swift://<hostname>[:<port>]/<container>[/<prefix>]
```

Keystone(v2 / v3)認証では:

```
swiftks://<hostname>[:<port>]/<region>:<container>[/<prefix>]
```

Keystone v3 認証を使うときは `domain` バックエンドオプション(後述)も指定する必要があります。

どちらの場合も *hostname* は認証サーバの名称です。ストレージコンテナは事前に存在している必要があります(OpenStack プロバイダはコンテナ作成用の Web フロントエンドや CLI を提供しています)。*prefix* は S3QL が使用するすべてのオブジェクト名に付加される任意のプレフィックスで、同一コンテナ内に複数の S3QL ファイルシステムを格納できます。

レガシー認証時は、バックエンドログイン/パスワードは OpenStack のユーザ名と API アクセスキーに対応します。Keystone 認証時は、バックエンドパスワードは通常の OpenStack パスワード、バックエンドログインは `<tenant>:<user>` の形式で OpenStack ユーザ名とテナント/プロジェクトを結合したものです。テナントが不要なら OpenStack ユーザ名だけでも構いません。Keystone v2 では `<tenant>` はテナント名(OpenStack RC ファイルの `OS_TENANT_NAME`)、Keystone v3 ではプロジェクト ID(`OS_TENANT_ID`)です。

OpenStack バックエンドは次のオプションを受け付けます:

- `no-ssl`: 認証サーバへの接続に平文 HTTP を使用。ストレージサーバへの接続に HTTPS か HTTP を使うかは認証サーバが決定するため、このオプションは直接影響しません。
- `ssl-ca-path=<path>`: 指定 CA でサーバ証明書を検証。
- `tcp-timeout`: TCP タイムアウト(デフォルト: 20 秒)。
- `disable-expect100`: データアップロード時に `Expect: continue` ヘッダを使用しません([RFC2616 8.2.3 節](http://tools.ietf.org/html/rfc2616#section-8.2.3))。HTTP 1.1 を完全にサポートしない壊れたストレージサーバへの回避策として使えますが、場合によってはデータが複数回送信されるため性能が低下することがあります。
- `no-feature-detection`: Swift バックエンドの高度機能を動的検出しません。この場合、S3QL はサポートされる Swift のバージョンや設定の最小公倍数機能しか使用できません。
- `domain`: Keystone v3 API を使います。OpenStack のデフォルトドメイン ID は `default` です。`project-domain` を指定しない場合、このドメインがプロジェクトとユーザ双方のドメインとして使われます。ドメイン名ではなくドメイン ID を指定します。プロバイダがドメイン ID を提供していなければ `default` である可能性が高いです。
- `domain-is-name`: プロバイダがドメイン名のみ(UUID なし)を提供する場合、このオプションを設定し `domain` をドメイン名として使用します。
- `project-domain`: 単純な場合、プロジェクトドメインは認証ドメインと同じです。指定しない場合はユーザドメインと同じと仮定されます。ドメイン ID を指定します。
- `project-domain-is-name`: プロジェクトドメインを名前指定する場合に設定。指定がなければ `domain-is-name` と同じと仮定されます。
- `tenant-is-name`: プロバイダによってテナント指定に名前を使うか ID を使うか異なるため、名前を使う場合はこのオプションを設定します。
- `identity-url`: プロバイダが `hostname:port/v3/auth/tokens` 以外の認証 URL(例 `/identity/v3/auth/tokens` など)を使用する場合に設定できます。

### Rackspace CloudFiles

[Rackspace](http://www.rackspace.com/) CloudFiles は内部で [OpenStack](http://www.openstack.org/) を使用しているため、ホスト名に `auth.api.rackspacecloud.com` を指定すれば上述の OpenStack/Swift バックエンドがそのまま使えます。利便性のため、専用の `rackspace` バックエンドも用意されています:

```
rackspace://<region>/<container>[/<prefix>]
```

ストレージコンテナは指定リージョンに事前作成されている必要があります。*prefix* はここでも同様の意味で使えます。

コンテナは [Cloud Control Panel](https://mycloud.rackspace.com/) で作成できます(上部メニューの *Files* をクリック)。

Rackspace バックエンドは OpenStack バックエンドと同じオプションを受け付けます。

### S3 互換ストレージ

S3 互換バックエンドは、Amazon S3 と同じプロトコルを使用する任意のストレージサービスにアクセスできます。ストレージ URL:

```
s3c://<hostname>:<port>/<bucketname>/<prefix>
```

`s3c://` は API リクエスト認証に AWS V2 署名を使います(AWS 側では廃止予定ですが、多くの互換サービスでは引き続き受け入れられています)。

`s3c4://` は同じバックエンドの別バージョンで、AWS V4 署名を使います:

```
s3c4://<hostname>:<port>/<bucketname>/<prefix>
```

S3 互換バックエンドは次のオプションを受け付けます:

- `no-ssl`: 平文 HTTP を使用。
- `ssl-ca-path=<path>`: 指定 CA で検証。
- `tcp-timeout`: TCP タイムアウト(デフォルト: 20 秒)。
- `disable-expect100`: `Expect: continue` ヘッダを使用しません。
- `dumb-copy`: サーバが `200 OK` を返した時点で COPY リクエストが成功したと仮定します。[S3 COPY API](http://docs.aws.amazon.com/AmazonS3/latest/API/RESTObjectCOPY.html) の仕様では、サーバは成功ステータスを返したあとで本文にエラーを返す可能性があるため、このオプションはサーバが正しい応答本文を返さない場合にのみ使用してください。
- `sig-region=<region>`: `s3c4://` 専用。V4 リクエスト署名計算に使うリージョンです(S3 と異なりストレージ URL に含まれないため別途指定が必要)。デフォルトは `us-east-1`。

### Backblaze B2

Backblaze B2 は独自の API を持つクラウドストレージです。

> **警告**: S3QL 開発者は Backblaze インスタンスへのアクセス権がなく、リリース前テストが行われないため、このバックエンドは予期せず壊れることがあります。Backblaze ユーザがテスト・保守に貢献(動作しなければ PR を提出)してくれることに依存しています。

ストレージ URL:

```
b2://<bucket-name>[/<prefix>]
```

*bucket-name* は提供されたアカウントキーでアクセス可能な既存のバケットです。*prefix* はいつも通り、1 つのバケット内に複数のリポジトリを保持するのに使えます。

アプリケーションキーも使用可能です。必要なキー権限は次のとおりです:

- `listBuckets`
- `listFiles`
- `readFiles`
- `writeFiles`
- `deleteFiles`

オプション:

- `disable-versions`: バケットのバージョニングが無効な場合に設定。オブジェクト削除時にすべてのファイルバージョンをスキャンせず、最新バージョンのみ存在するとみなすことで、クラス C トランザクションの代わりに 1 回のクラス B トランザクションで済ませます。
- `retry-on-cap-exceeded`: Backblaze アカウントにデータ/トランザクション上限があるとき、上限カウンタは毎日リセットされるため、このオプションで操作を再試行するか制御します。設定しない場合は例外でプログラムが中断します。
- `test_mode=<value>`: 特殊ヘッダを送信してサーバをテストモードにします。失敗耐性のテスト専用で、無駄なトラフィック、遅延、トランザクションを発生させます。有効値の例: `fail_some_uploads`、`expire_some_account_authorization_tokens`、`force_cap_exceeded`(詳細は [Backblaze ドキュメント](https://www.backblaze.com/docs/en/cloud-storage-integration-checklist))。
- `tcp-timeout`: TCP タイムアウト(デフォルト: 20 秒)。

### ローカル

S3QL はデータをローカルファイルシステムに保存することもできます。外部メディアへのバックアップや、S3QL が直接対応しない外部サービスへのアクセス(たとえば [sshfs](http://fuse.sourceforge.net/sshfs.html) でリモートシステムをマウントし、その上にローカルバックエンドを重ねる)に使えます。

ストレージ URL:

```
local://<path>
```

絶対パス指定にはスラッシュを 3 つ連続させる必要があることに注意してください(例: `local:///var/archive`)。また、相対パスは認証ファイル読み込み前に自動的に絶対パスへ変換されます。たとえば `/home/john` で `local://s3ql` をマウントしようとした場合、認証ファイルのセクションは `local:///home/john/s3ql` と一致する必要があります。

ローカルバックエンドはオプションを受け付けません。

## データ損失を防ぐための重要なルール

S3QL のバックエンドの多くは分散ストレージシステムにデータを保存します。これらは従来のローカルハードディスクとはいくつか重要な点で異なるため、データ損失を避けるためにこの節は注意深く読んでください。

### ルールの要点

1. 選択するストレージプロバイダから期待できる **耐久性** を把握すること。耐久性は、保存されたオブジェクトが時間経過で損傷する可能性を表します。データ破損を完全に防ぐことは不可能で、地理的複製や RAID といった技術はその可能性を低下させる(耐久性を高める)だけです。
2. バックエンドとプロバイダを選ぶ際、S3QL のデータ重複排除機能によりファイルシステムデータの実効耐久性が低下することを念頭に置いてください。
3. プロバイダが読み書き・一覧取得について **即時整合性(immediate consistency)** を提供することを確認してください。**結果整合性(eventual consistency)** は S3QL には不十分で、完全なデータ損失につながる可能性があります(結果整合性は可用性向上と引き換えに、変更が即座に反映されない可能性があります)。
4. 同じファイルシステムを複数のコンピュータで同時にマウントしないこと(同じコンピュータでも異なる `--cachedir` を使う場合も同様)。
5. 正常にアンマウントされていないファイルシステムを、別のコンピュータ(または別の `--cachedir`)でマウントまたは fsck しないこと。

### データ耐久性

耐久性は、保存されたオブジェクトが時間経過で破損する確率の指標です。データ損失の可能性が低いほど耐久性が高いといえます。Amazon S3 のようなサービスは年間 99.999999999% の耐久性をうたっており、これは 100,000,000 個のオブジェクトを 100 年間保存しても高々 1 個しか失われないレベルです。

S3QL は冗長性を減らし、可能な限り小さな形でデータを保存するよう設計されています。そのため S3QL は通常、ストレージ側の損失を補償できません。選定時には提供される耐久性が要件に見合うか慎重に確認してください。

2 つの点に注意が必要です。

**第一に**、S3QL はバックエンドの障害を補償できませんが、**検知は可能** です。失われたり破損したデータにアクセスしようとすると IO エラーが返されます。

**第二に**、重複排除機能により、時点 *x* でストレージ側がデータを失うと、時点 *x* 以降に書き込んだデータまで失われる可能性があります:

1. 重要なファイルを S3QL に保存。
2. ストレージ側がそのデータブロックを失う。アクセスするか **s3ql_verify** を実行しない限り、S3QL はこれに気付かない。
3. 同じ S3QL ファイルシステムの別の場所に同じファイルをコピー。
4. S3QL は新しいファイルの内容が既存ブロックと同じだと検出し、重複排除のため新規アップロードせず既存オブジェクトへのリンクとして扱う。
5. 結果、コピーを作ったつもりでも、両方が失われたブロックを参照しているためバックアップになっていない。

一部のストレージサービスでは **fsck.s3ql** でこれを緩和できます。fsck.s3ql はストレージにオブジェクト一覧を問い合わせ、欠損があれば該当ファイルにマークを付けて問題の波及を防ぎます。比喩的にいえば「チェックポイント」を設けるイメージで、fsck 以前に発生した損失が fsck 以降の操作に波及しません。ただし、多くのサービスはオブジェクトが実際に取得されるまで欠損を検知しないため、一覧問い合わせだけでは **fsck.s3ql** には分からないことがあります。

さらに **s3ql_verify** コマンドを併用すれば、全オブジェクトを実際に問い合わせてストレージの欠損/破損を確認できます。実行時間は長くかかりますが、より強力な保証が得られます。s3ql_verify が検出した破損オブジェクトから「回復」するには、破損オブジェクトを明示的に削除してから **fsck.s3ql** を実行し、ファイルシステムメタデータを修正させ、影響を受けたファイルを `lost+found` に移動させる必要があります。この手順は現状自動化されていないため、長期にわたって欠損オブジェクトの可能性を無視できるほど耐久性が高いサービスを選ぶのが賢明です。

## ファイルシステムの作成

S3QL ファイルシステムは **mkfs.s3ql** コマンドで作成します。構文:

```
mkfs.s3ql [options] <storage url>
```

主なオプション:

- `--cachedir <path>`: キャッシュデータ格納ディレクトリ(デフォルト: `~/.s3ql`)
- `--log <target>`: ログ出力先。`none` で標準出力、`syslog` でシステムロギングデーモン、それ以外はファイル名。1 MiB 超でローテート、最大 5 世代保持。デフォルト `None`
- `--debug-modules <modules>`: 指定モジュールのデバッグ出力を有効化(カンマ区切り、`all` で全体)
- `--debug`: 全モジュールのデバッグ出力を有効化
- `--quiet`: 極力静かに動作
- `--backend-options <options>`: バックエンド固有オプション(カンマ区切り)
- `--version`: バージョン表示のみ
- `--authfile <path>`: 認証情報ファイル(デフォルト: `~/.s3ql/authinfo2`)
- `-L <n>`: ファイルシステムラベル
- `--data-block-size <size>`: ファイル内容のブロックサイズ(KiB)。これを超えるファイルは複数オブジェクトに分割。圧縮により実オブジェクトはさらに小さくなることも。デフォルト 1024 KiB
- `--metadata-block-size <size>`: メタデータ用ブロックサイズ(KiB)。デフォルト 64 KiB
- `--plain`: 暗号化なしのファイルシステムを作成

`--plain` を指定しない限り、暗号化パスフレーズの入力を求められます。暗号化ファイルシステムが誤って作成されるのを防ぐため、このパスフレーズは `--authfile` からは読み込まれません。

注意事項:

- 指定したストレージ URL 配下は S3QL が専有管理するものとして扱われます。手動でオブジェクトを追加・削除・変更するとファイルシステムが壊れ、**fsck.s3ql** がそれらを削除する可能性があります。
- 多くのバックエンドでは URL プレフィックス中のスラッシュに特別な意味はありません。例えば `s3://mybucket/myprefix/` と `s3://mybucket/myprefix` は別物です。
- S3QL ファイルシステムは「入れ子」にできません。たとえば `s3://bucketname/outerprefix` にファイルシステムを作り、その内側 `s3://bucketname/outerprefix/innerprefix` に別のファイルシステムを作ることはできません。

## ファイルシステムの管理

`s3qladm` コマンドはアンマウント状態の S3QL ファイルシステムに対して各種操作を行います。マウント中は **絶対に使用しないでください**。深刻な問題を引き起こします。

構文:

```
s3qladm [options] <action> <storage-url>
```

`action` は **passphrase**、**restore-metadata**、**clear**、**recover-key**、**upgrade** のいずれか。

共通オプションは `--authfile`、`--debug-modules`、`--debug`、`--quiet`、`--log`、`--backend-options`、`--cachedir`、`--version` です(mkfs.s3ql と同義)。

ヒント: 各アクションの追加引数は `s3qladm <action> --help` で確認できます。

### パスフレーズの変更

```
s3qladm passphrase <storage url>
```

これは保存済みデータを新パスフレーズで再暗号化するわけではありません。S3QL はファイルシステム作成時に内部的に「マスターキー」を生成し、そのキーで全データを暗号化します。ユーザーのパスフレーズはマスターキーの暗号化/復号にのみ使われるため、いつでも変更可能です。ただし、**過去にパスフレーズが漏洩していた場合、変更しても元のパスフレーズを持つ者のアクセスを取り消せません**(彼らは既にマスターキーを復号してコピーしている可能性があるため)。

### マスターキーの復元

極めて不運なことに、S3QL のバグやプロバイダのデータ損失でマスターキーを保持するオブジェクトにアクセスできなくなった場合、パスフレーズを知っていても復号できません。

幸いファイルシステム作成時に **mkfs.s3ql** が作成を促した外部コピーがあれば、それを使ってマスターキーを復元できます:

```
s3qladm recover-key <storage url>
```

### メタデータバックアップの復元

最新のメタデータが修復不能なほど破損した場合、自動作成されたバックアップを使って復元できます:

```
s3qladm restore-metadata <storage url>
```

利用可能なメタデータバックアップの一覧が表示され、どれを復元するか選択できます。

> **警告**: この機能は、まずメーリングリストで相談してから使うべきです(「追加リソース」参照)。

### ファイルシステムの削除

```
s3qladm clear <storage url>
```

全データとファイルシステム構造を物理的に削除します。

### 未使用領域の回収

ファイルやディレクトリを削除したとき、メタデータデータベース内の対応領域はすぐには解放されず、空きとしてマークされて新しいエントリで再利用されます。この領域を回収して、ディスク上およびバックエンド上のメタデータファイルサイズを縮小するには:

```
s3qladm shrink-db <storage url>
```

### ファイルシステムのアップグレード

新バージョンの S3QL をインストールした際、ファイルシステムのメタデータもアップグレードが必要な場合があります。アップグレード後は古いバージョンではアクセスできなくなります。

アップグレード中にコマンドが中断されたり、同時にマウント/チェック/アップグレードが行われないようにしてください:

```
s3qladm upgrade <storage url>
```

## マウント

S3QL ファイルシステムは **mount.s3ql** でマウントします:

```
mount.s3ql [options] <storage url> <mountpoint>
```

> **注記**: S3QL は NFS や CIFS のようなネットワークファイルシステムではありません。一度に 1 台のコンピュータでしかマウントできません。

主なオプション:

- `--log <target>`: ログ出力先(デフォルト: `~/.s3ql/mount.log`)
- `--cachedir <path>`: キャッシュディレクトリ(デフォルト: `~/.s3ql`)
- `--debug-modules <modules>`、`--debug`、`--quiet`、`--backend-options`、`--version`、`--authfile`: 上記コマンドと同義
- `--compress <algorithm-lvl>`: 圧縮アルゴリズムとレベル。アルゴリズムは `lzma`、`bzip2`、`zlib`、`none`。レベルは 0(最速)〜9(最遅)。デフォルト `lzma-6`
- `--cachesize <size>`: キャッシュサイズ(KiB、デフォルト: 自動検出)
- `--max-cache-entries <num>`: キャッシュ最大エントリ数(デフォルト: 自動検出)。各エントリは 1 ファイルディスクリプタを消費するため、増やす場合は `ulimit -n` で fd 上限を十分に(エントリ数+100 以上)確保してください
- `--keep-cache`: 終了時にローカルキャッシュを削除しない
- `--allow-other`: 通常はマウントしたユーザーのみアクセス可能ですが、このオプションで他ユーザーにもアクセスを許可し、ファイルごとのパーミッションに従わせます
- `--allow-root`: `--allow-other` と似ていますが、マウントユーザーと root のみに制限
- `--dirty-block-upload-delay <seconds>`: ダーティブロックのアップロード遅延(デフォルト: 10 秒)
- `--fg`: デーモン化せずフォアグラウンドで動作
- `--fs-name FS_NAME`: FUSE に渡すマウント名(`mount` コマンド出力の先頭列に表示)。省略時はストレージ URL を使用
- `--systemd`: systemd unit として実行。`--log none` と組み合わせて journald を利用するのがおすすめ
- `--metadata-backup-interval <seconds>`: メタデータバックアップ間隔。マウント中にクラッシュすると最終バックアップ以降の変更は失われる可能性があります。バックアップ中はファイルシステムが応答不能になります。デフォルト: 6 時間
- `--max-connections <no>`: 並列アップロード接続数(デフォルト: 32)
- `--max-threads <no>`: 並列圧縮/暗号化スレッド数(デフォルト: 自動)
- `--nfs`: NFS エクスポート用の最適化を有効化(デフォルト: False)

### 権限チェック

`--allow-root` も `--allow-other` も指定しない場合、マウントしたユーザーはファイル所有者やモードに関わらず全権限を持ちます。どちらかを指定した場合は標準的な UNIX パーミッションチェックが行われ、実際の root のみ完全アクセス可、他のユーザー(マウントユーザー含む)はチェック対象になります。

### 圧縮アルゴリズム

S3QL は LZMA、Bzip2、zlib の 3 つをサポート(デフォルトは LZMA)。圧縮は新規ブロックにのみ影響するため、マウントのたびに自由に切り替え可能です。

おおまかには、LZMA は遅いが圧縮率が高く、Bzip2 はその中間、zlib は速いが圧縮率が低めです。

最適なアルゴリズムはネットワーク速度に依存します。圧縮はネットワーク帯域を飽和させるのにちょうど足りる速度が理想です。

最適化のために、`contrib` ディレクトリに `benchmark.py` が同梱されています。ファイルシステムのブロックサイズに近いサイズで類似の内容のファイルに対して実行し、異なるアルゴリズムの圧縮速度と指定バックエンドのアップロード速度を計測して推奨を出してくれます。実行時は他の負荷を避けてください(動画エンコードやコンパイルをしながら走らせないこと)。

### キャッシュに関する注意

S3QL はアクセス高速化のためローカルキャッシュを保持します。キャッシュはブロック単位なので、ファイルの一部だけがキャッシュ内にある場合もあります。

#### キャッシュエントリ数の上限

キャッシュサイズは `--cachesize` で、オブジェクト数は `--max-cache-entries` で制限します。各エントリは 1 つの fd を使うため、通常の Linux 上限(約 1000)に制約されます。`--max-cache-entries` を増やすなら `ulimit -n` も併せて調整してください。

#### キャッシュのフラッシュと期限切れ

変更されたブロックは、ある程度変更されなかった状態が続くか、キャッシュが溢れそうになった時点でバックエンドに書き込まれます。この時間は `--dirty-block-upload-delay` で指定できます。

キャッシュの期限切れ(削除)は最大キャッシュサイズに達したときのみ行われ、LRU(最も使われていないブロック)順に削除されます。

### NFS サポート

S3QL ファイルシステムは NFS 経由でエクスポートできます。NFS 利用時は `--nfs` オプションでパフォーマンス改善が期待できますが、指定しなくても害はありません。

NFS はサーバ再起動を跨いだクライアントマウントの持続性をサポートします。`fsck.s3ql` を実行しない限り、S3QL でもサーバ側でのアンマウント/再マウント(あるいは再起動)後もクライアントは一時的な不可達だけで影響を受けません。

> **警告**: `fsck.s3ql` がファイルシステムを変更した場合、サーバで S3QL を再マウントする前に、すべての NFS クライアントを一度アンマウントしてから再マウントし直す必要があります。

### 障害モード

マウント後のリモートサーバ通信では多種多様な障害が起こり得ます。mount.s3ql は状況下で可能な限りアクセス可能な状態を維持しようとします。ネットワーク切断時でも、ローカルキャッシュに空きがある限り書き込みは可能です。キャッシュにないデータを読む操作は接続復旧までブロックされます。データ破損を検知した場合はリードオンリーに切り替わり、影響ファイルを読もうとすると入出力エラー(errno が `EIO`)が返ります。

その他の予期せぬ致命的問題では mount.s3ql は終了しますが、ファイルシステムはアンマウントされません。アクセスすると「Transport endpoint not connected」エラー(errno が `ESHUTDOWN`)が返ります。これは mount.s3ql が終了した状態のマウントポイントを、空のファイルシステムと取り違えないようにするためです(例えばマウントポイントが自動ミラーされている場合、取り違えは致命的です)。この状態のマウントポイントは FUSE の **fusermount -u** で解除できます。

mount.s3ql はネットワーク切断や一時的サーバエラー時には自動で再接続・再試行します。再試行間隔は 20 ms から最大 5 分まで増やされ、最大 24 時間試行が続きます。24 時間後には諦めて終了し、マウントポイントは上述の状態になります。

mount.s3ql は異常事象を検出するたびにログを出します。出力先は `--log` で指定できます。[logcheck](http://sourceforge.net/projects/logcheck/) などで定期チェックすることを強く推奨します。アクセス制限には至らないものの調査すべき事象が多数あり、ログ以外では気付けないためです。

### 自動マウント

システム起動/停止時に自動マウント/アンマウントしたい場合は、`/etc/fstab` ではなく専用の init ジョブを使ってください。systemd を使う場合、`--systemd` オプションで mount.s3ql を `notify` タイプのサービスとして実行できます。

`/etc/fstab` に適切なエントリを書けば自動マウントは原理的に可能ですが、次の理由で推奨されません:

- `/etc/fstab` のファイルシステムは **umount** で解除されますが、これは全データアップロードを待たずシャットダウン/再起動が進みます(FUSE 制約)。
- S3QL マウントに Python インタプリタが必要であることをシステムに伝える方法がなく、Python を含むボリュームがマウントされる前に mount.s3ql が実行される可能性があります。
- インターネット接続が利用可能になってから S3QL をマウントするよう伝える標準的方法もありません。

## S3QL の高度な機能

### スナップショットとコピーオンライト

`s3qlcp` コマンドで、実データを物理的にコピーせずにディレクトリツリーを複製できます。これはデータ重複排除機能によって可能になります:

```
s3qlcp [options] <src> <target>
```

`<src>` は既存のディレクトリで、`<target>` は存在しないこと、かつ両者は同一 S3QL ファイルシステム内にあることが必須です。

複製でファイル内容分の容量は増えません。メタデータ使用量はエントリ数とアイノード数に比例して増加します。後でどちらかが変更されたとき、変更部分のみが新たな容量を消費します。

`s3qlcp` はマウントしたユーザー(および `--allow-other`/`--allow-root` 指定時は root)のみ実行可能です。

複製後の `<src>` と `<target>` はどちらも完全に普通のディレクトリです。片方を他方のスナップショットと見なせます。典型的な使い方は、`documents` を毎月 `documents_January`、`documents_February` などに複製する運用です。

同じ結果は `cp -a` でも得られますが、S3QL では比較にならないほど遅くなります。`cp` は全データをネットワーク越しに読み込み、書き込み時に S3QL が重複排除することになるためです。

#### スナップショット vs ハードリンク

S3QL のスナップショット機能は、[rsync](http://www.samba.org/rsync) や [storeBackup](http://savannah.nongnu.org/projects/storebackup) などが使うハードリンク機能に影響を受けていますが、ハードリンクには欠点があります:

- バックアップ・リストアには専用プログラムが必要で、他のプログラムで触れると他のハードリンク先に意図せず影響します。
- すでにハードリンクされたファイルを特別扱いする必要があります(バックアッププログラムが作成したものかどうか判別する必要あり)。

S3QL のスナップショットはこうした問題がなく、任意のバックアッププログラムと組み合わせて使えます。

### 統計情報の取得

マウント中のファイルシステムの情報は `s3qlstat` で確認できます:

```
s3qlstat [options] <mountpoint>
```

出力例:

```
Directory entries:    1488068
Inodes:               1482991
Data blocks:          87948
Total data size:      400 GiB
After de-duplication: 51 GiB (12.98% of total)
After compression:    43 GiB (10.85% of total, 83.60% of de-duplicated)
Database size:        172 MiB (uncompressed)
(一部の値にはキャッシュ内の未アップロードダーティブロックが反映されません)
```

とくに興味深いのは、総データサイズ、重複排除後のサイズ、さらに圧縮後の最終サイズです。

`s3qlstat` はマウントしたユーザー(+ `--allow-other`/`--allow-root` 時 root)のみ実行可能です。

### イミュータブルツリー

**s3qllock** で指定ディレクトリツリーを変更不可にできます。追加・削除・変更が一切できず、削除するには後述の **s3qlrm** のみ使えます:

```
s3qllock 2010-04-21
```

これはバックアップ向け機能です。従来のテープバックアップでは、取った後にテープを取り外して保管すれば、物理破壊以外で内容は変わりません。

一方、S3QL のようなオンラインストレージでは、マウント中いつでもバックアップにアクセスできてしまい、古いバックアップの中のファイルを後から変更することも可能です。最悪の場合、バックアップシステム全体が無意味になります。ウィルスに感染して見つけたファイルを全部削除する場合、マウント中なら元データと共にバックアップも破壊されます。

マルウェアの有無に関わらず、作成後のバックアップを変更できるのは望ましくありません。よくある使い方として、S3QL を常時マウントし定期的に `rsync -a` でバックアップ、各ユーザーが自分で過去ファイルを復元する運用がありますが、この場合誤って古いバックアップ内のファイルを変更・削除してしまうことも起こり得ます。

イミュータブル化はこれらを防ぎます。S3QL を狙ったウィルスでもない限り、一度イミュータブル化したバックアップは変更も削除もできません。

### 高速再帰削除

`s3qlrm` は S3QL 上のファイル・ディレクトリを再帰的に削除します。`rm -r` より高速ですが、存在意義は主に「イミュータブルツリーも削除できる」ことにあります:

```
s3qlrm <directory>
```

追加確認はなく、即座に削除されるので注意してください。

### 実行時の設定

```
s3qlctrl [options] <action> <mountpoint> ...
```

s3qlctrl でマウント中のファイルシステムに対して各種操作を行います。マウントユーザー(+ `--allow-other`/`--allow-root` 時 root)のみ実行可能です。

アクション:

- **flushcache**: すべての変更済みブロックをバックエンドに書き出し、完了までブロック
- **dropcache**: キャッシュをフラッシュしてから全破棄(空に)
- **log**: ログレベル変更  
  `s3qlctrl [options] log <mountpoint> <level> [<module> ...]`  
  level は `debug`、`info`、`warn`。module は debug レベルでのみ指定可能で、デバッグ出力を特定モジュールに絞り込みます
- **cachesize**: キャッシュサイズ変更  
  `s3qlctrl [options] cachesize <mountpoint> <new-cache-size>`(KiB 単位)
- **backup-metadata**: メタデータバックアップを実行

## アンマウント

```
umount.s3ql [options] <mountpoint>
```

全データがバックエンドに書き込まれるまでブロックします。

**mount.s3ql** でマウントしたユーザーのみアンマウント可能です。root が他ユーザーのマウントを解除するには **fusermount -u** または **umount** を使う必要があります(これらは全データのアップロード完了を待たずすぐ返るため、使う場合は `mount.s3ql` プロセスが終了するのを手動で待ってからシャットダウンしてください)。

umount.s3ql のオプション:

- `--log`、`--debug-modules`、`--debug`、`--quiet`、`--version`: 共通オプション
- `--lazy, -z`: レイジーアンマウント。開いているファイルがあっても即座にファイルシステムを切り離し、バックグラウンドで全ファイルが閉じられた後にアップロードを完了させます

`umount.s3ql` が何らかの理由で動かない場合、`fusermount -u -z` でもアンマウント可能です。ただしこれは即座に返り、バックグラウンドでアップロードが続くことがあります。

## エラーチェック

**fsck.s3ql** と **s3ql_verify** を(この順で)定期的に実行し、ファイルシステムの一貫性、バックエンドのデータ破損/損失の有無を確認することが推奨されます。

**fsck.s3ql** はクラッシュや S3QL のバグによる内部構造問題の検出・修正が目的です。バックエンドは完全に信頼できるものと仮定し、「存在する」と報告されれば実体も健全とみなします。

対して **s3ql_verify** はバックエンドの整合性が目的で、内部メタデータは正しいと仮定し、全データが実際に取得可能かを検証します。fsck.s3ql より時間がかかる場合があります。

### 内部ファイルシステムエラーのチェックと修復

S3QL が正常にアンマウントされなかった場合、再マウント前に fsck.s3ql を実行する必要があります:

```
fsck.s3ql [options] <storage url>
```

主なオプション:

- `--log`、`--cachedir`、`--debug-modules`、`--debug`、`--quiet`、`--backend-options`、`--version`、`--authfile`、`--compress`、`--keep-cache`: 既出
- `--batch`: ユーザー入力が必要な場合、プロンプトせず終了
- `--force`: clean マークがあっても強制チェック
- `--force-remote`: リモートメタデータの使用を強制(データ損失につながる可能性あり)
- `--fast`: より高速で簡易なチェック
- `--max-threads <no>`: 並列スレッド数(デフォルト: 自動)

### バックエンドデータ破損の検出と対処

**s3ql_verify** はバックエンドのオブジェクトリストを信頼せず、実際に各オブジェクトへの取得を試みます。デフォルトでは各オブジェクトのメタデータのみ取得(S3 互換系なら HEAD リクエスト)して存在確認します。`--data` を指定すると実データを全部読み込みます(このとき転送量は s3qlstat の「After compression」行で確認できます)。

s3ql_verify は破損データを修復せず、破損/欠損オブジェクトのリストをファイルに書き出すだけです。対応はユーザー判断に委ねられます。バックエンド管理者権限があるなら原因調査やバックアップからの復元を試みてください。完全に失われたと判断したら、`contrib` ディレクトリの [remove_objects.py](http://www.rath.org/s3ql-docs/contrib.html#remove-objects) で対象オブジェクトを明示削除できます。その後 **fsck.s3ql** を実行すれば、削除済みのオブジェクトが欠損として検出され、メタデータが更新され、影響を受けたファイルは `lost+found` へ移動されます。

```
s3ql_verify [options] <storage url>
```

主なオプション:

- `--log`、`--debug-modules`、`--debug`、`--quiet`、`--version`、`--cachedir`、`--backend-options`、`--authfile`: 既出
- `--missing-file <n>`: 欠損オブジェクトのキー保存先ファイル
- `--corrupted-file <n>`: 破損オブジェクトのキー保存先ファイル
- `--data`: メタデータだけでなくオブジェクト全体を読み込み
- `--parallel PARALLEL`: 並列接続数
- `--start-with <n>`: 最初の n 個をスキップし、n+1 番目から検証

## バックエンド情報と認証情報の保存

通常 S3QL は認証情報や暗号化パスフレーズを端末から読み取りますが、`~/.s3ql/authinfo2`(または `--authfile` 指定ファイル)で指定することもできます。

authinfo ファイルは `[section]` ヘッダと `name: value` エントリで構成されます。セクションヘッダ自体は S3QL では使用されませんが(`[<backend>-test]` の特殊セクションを除く)、ファイル内で一意である必要があります。

各セクションで定義できるエントリ:

- `storage-url`: このセクションが適用されるストレージ URL。URL がここで指定した文字列で始まれば適用されます
- `backend-login`: バックエンド認証用ユーザ名
- `backend-password`: バックエンド認証用パスワード
- `fs-passphrase`: ファイルシステム復号用パスフレーズ(暗号化時)

ストレージ URL を扱うコマンド共通のコマンドラインパラメータも、先頭のダッシュを省く形で記述可能です(例: `backend-options: notls`)。

読み込み時は該当セクションを順に見て、各エントリの最後に見つかった値を採用します。例:

```
[s3]
storage-url: s3://
backend-login: joe
backend-password: notquitesecret

[fs1]
storage-url: s3://joes-first-bucket
fs-passphrase: neitheristhis

[fs2]
storage-url: s3://joes-second-bucket
fs-passphrase: swordfish

[fs3]
storage-url: s3://joes-second-bucket/with-prefix
backend-login: bill
backend-password: bi23ll
fs-passphrase: ll23bi
```

この例では、s3 バックエンド使用時は原則 `joe` としてログインしますが、`s3://joes-second-bucket/with-prefix` で始まる URL では最後のセクションが有効になり `bill` の認証情報が使われます。暗号化パスフレーズも `s3://joes-first-bucket` や `s3://joes-second-bucket` で始まる URL で使われます。

このファイルは [Python ConfigParser モジュール](http://docs.python.org/library/configparser.html) でパースされます。

## 寄稿プログラム

S3QL にはコア配布物には含まれない(通常自動インストールもされない)寄稿プログラムが付属しています。ソース配布物の `contrib` ディレクトリ、またはパッケージインストール時は `/usr/share/doc/s3ql/contrib` にあります。

### benchmark.py

S3QL の書き込み性能、アップリンク帯域、圧縮速度を計測し、ボトルネックを特定します。圧縮アルゴリズムと並列アップロード接続数の推奨も出してくれます。

### pcp.py

`pcp.py` は複数の rsync プロセスを起動して並列にディレクトリツリーをコピーするラッパーです。S3QL からのコピー性能は並列化で大きく改善するため重要です(「コピー性能の向上」参照)。

`/mnt/home-backup` を 8 並列で `/home/joe` にパーミッション保持コピー:

```
pcp.py -a --processes=8 /mnt/home-backup/ /home/joe
```

### s3ql_backup.sh

S3QL と [rsync](http://samba.org/rsync) を使ったシンプルで強力なバックアップ手順のサンプルスクリプトです。次の手順を自動化します:

1. ファイルシステムをマウント
2. 前回バックアップを [s3qlcp](http://www.rath.org/s3ql-docs/special.html#s3qlcp) で複製
3. rsync でバックアップ元から新コピーを更新
4. 新バックアップを [s3qllock](http://www.rath.org/s3ql-docs/special.html#s3qllock) でイミュータブル化
5. 不要な古いバックアップを削除
6. アンマウント

バックアップは `YYYY-MM-DD_HH:mm:SS` 形式のディレクトリに保存され、古いものの削除には expire_backups.py を使います。

### expire_backups.py

古いバックアップを賢く削除するプログラムです。

「どのバックアップをどれくらいの期間残すか」を **年代範囲(age range)** で定義します。各年代範囲に常に 1 つ以上のバックアップが存在することを保証し、それに必要な分だけ残して他は削除します。

年代範囲はバックアップサイクル数で指定します。新しいバックアップを作るたびに既存バックアップは 1 サイクル分「歳をとり」ます。

例: `1 3 7 14 31` を指定すると、次の条件が常に満たされます:

1. 0〜1 サイクル古いバックアップ(最新)
2. 1〜3 サイクル古いバックアップ
3. 3〜7 サイクル古いバックアップ
4. 7〜14 サイクル古いバックアップ
5. 14〜31 サイクル古いバックアップ

> **注**: 一定間隔でバックアップを取る場合、1 サイクル = 1 間隔です。年代範囲を日数や週数でなくサイクル数で指定する利点は、不規則なバックアップ間隔も柔軟に扱える点にあります。たとえば 1 ヶ月間 PC を使わなかった場合、絶対日数指定だと全バックアップが削除されかねません。

使い方は簡単で、`year-month-day_hour:minute:seconds`(`YYYY-MM-DD_HH:mm:ss`)形式のディレクトリとして保存されたバックアップに対し、カレントディレクトリで動作します:

```
expire_backups.py 1 3 7 14 31
```

S3QL 上のバックアップには `--use-s3qlrm` オプションも併用するのが良く、これにより削除に [s3qlrm](http://www.rath.org/s3ql-docs/special.html#s3qlrm) が使われます。

内部的には `.expire_backups.dat` という状態ファイルを使い、各バックアップが何サイクル前かを追跡します(ディレクトリ名の日付からは厳密な推定ができないため)。このファイルが壊れたり消えた場合は `--reconstruct-state` で再構築を試みられますが、バックアップ間隔がずっと一定でない限り正確さに欠けます。状態ファイルはなるべく触らないようにしてください。

### remove_objects.py

ストレージバックエンドから指定オブジェクトを削除するプログラムです。バックエンドレベルで動作するため、対象に S3QL ファイルシステムが存在する必要はありません。

## ヒントとコツ

### SSH バックエンド

S3QL のローカルバックエンドと [sshfs](http://fuse.sourceforge.net/sshfs.html) を組み合わせることで、任意の SSH サーバ上に S3QL ファイルシステムを作れます。まず sshfs でリモートディレクトリをマウント:

```
sshfs user@my.server.com:/mnt/s3ql /mnt/sshfs
```

そのマウントポイントを S3QL のローカル宛先に指定:

```
mount.s3ql local:///mnt/sshfs/myfsdata /mnt/s3ql
```

### バックアップファイルシステムの常時マウント

バックアップ用途なら常時マウントしておくと便利です(毎回マウント/アンマウントせず)。とくにサイズが大きい場合、1 ファイルだけ復元したいときのマウント/アンマウント時間を節約できます。

その場合は次を推奨:

- [s3qllock](http://www.rath.org/s3ql-docs/special.html#s3qllock) でバックアップ後にイミュータブル化
- バックアップ直後に [s3qlctrl backup-metadata](http://www.rath.org/s3ql-docs/special.html#s3qlctrl) で最新メタデータを安全に保存

### コピー性能の向上

> **注**: 以下は S3QL **から** コピーする場合のみで、S3QL **へ** のコピーには該当しません。

S3QL から小さなファイルを大量にコピーする(システム復元など)と、性能がかなり悪いと感じるはずです。

これは S3QL の仕組み上避けがたいものです。ファイル読み込みのたびにバックエンドからネットワーク越しに取得が必要で、ファイルサイズに関わらず最低限のネットワークレイテンシがかかります。小さいファイルを大量にコピーすると、時間の 99% はネットワーク応答待ちに費やされます。

理論上は「同時に複数ファイルをコピーする」だけで解決しますが、`cp`、`rsync`、`tar` などの UNIX ツールは 1 ファイルずつ直列処理します(ローカルディスクでは合理的)。

現状最良の回避策は、除外ルールで担当を分けて複数の rsync を同時起動することです。例(3 並列: a-f / g-l / それ以外、`+ */` で全ディレクトリに潜る):

```bash
#!/bin/bash

RSYNC_ARGS="-aHv /mnt/s3ql/ /home/restore/"

rsync -f "+ */" -f "-! [a-f]*" $RSYNC_ARGS &
rsync -f "+ */" -f "-! [g-l]*" $RSYNC_ARGS &
rsync -f "+ */" -f "- [a-l]*" $RSYNC_ARGS &

wait
```

最適並列数はネットワーク環境とファイルサイズ次第ですが、おおよそ 10 並列あたりが多くの場面で大幅な改善をもたらします。

S3QL 同梱の `pcp.py`(`contrib` ディレクトリ)を使えば、こうしたスクリプトを自分で書かなくても並列転送できます(「pcp.py」参照)。

## よくある質問(FAQ)

### ファイルシステムのスループットを改善するには?

3 つの律速要因があります:

1. プロセス〜カーネル〜S3QL 間のスループット
2. 圧縮速度
3. アップロード速度

`contrib/benchmark.py` で各スループットを測定し、推奨圧縮アルゴリズムとアップロードスレッド数を提示してくれます。

### なぜ `df` は 1 TB の空き容量を表示するのか?

S3QL のファイルシステムサイズは事実上無限ですが、`df` など多くの UNIX ツールは無限概念を扱えないため、S3QL は何らかの値を返さざるを得ません。そこで常に「50% 空き」かつ「少なくとも 1 TB 空き」と報告します。例えば 1.5 TB 保存していれば空きも 1.5 TB、2 MB しか保存していなければ残り空きは 1 TB と表示されます。

### どの OS が使えるか?

主に Linux で開発されていますが、必要な Python モジュールが揃っている FUSE 対応 OS ならば原理的に動作します。**umount.s3ql** は非 Linux では動かないトリックを使っている可能性がありますが、FUSE 付属の `fusermount -u` は常に使えます(違いは全データアップロード完了を待たない点だけ)。他 OS での成功/失敗をメーリングリストに報告してもらえると情報が増えて助かります。

### ファイルサイズに上限はあるか?

ありません。ストレージサービス側の最大オブジェクトサイズ制限は、S3QL が自動で透過的にファイルを小さなブロックに分割するため問題になりません。

### 巨大ファイルを少しだけ変更すると、全体のダウンロード/再アップロードになるのか?

いいえ。S3QL はファイルを設定可能なサイズのブロック(デフォルト 10 MB)に分割するので、変更部分を含む 1 ブロックのみ転送されます。

### 重複排除がよく分かりません...

重複排除のすばらしさは「ユーザーが何も知らなくても恩恵を受けられる」ことにあります。ユーザーからは通常のファイルシステムと区別できませんが、バックエンドとのやり取りでは同じデータを 1 度だけ保存し、容量を節約します。

### 「Transport endpoint not connected」エラーとは?

ファイルシステムがクラッシュしています。`mount.log` と `mount.s3ql_crit.log` を確認し、適切ならバグ報告してください。マウントポイントを再度使用可能にする(クラッシュしたファイルシステムをアンマウントする)には `fusermount -u` を使います。

バグ報告前に最新版の S3QL を使っていることを確認してください。

### 「Backend reports that fs is still mounted elsewhere, aborting」とは?

ファイルシステムがきれいにアンマウントされなかったか、まだマウント中であることを意味します。マウントされていないと確信できる場合は、直近にマウントしたコンピュータで **fsck.s3ql** を実行してください。

### 複数のコンピュータで同時にアクセスできるか?

できません。複数台で共有するには「いつデータを変更したか」を互いに通知する仕組みが必要ですが、S3QL が対象とするクラウドストレージでは、異なるクライアントが異なるサーバと通信する可能性があり、十分な頻度での同期を強制できません。

したがって 1 台を「マスター」として mount.s3ql を実行させ、そのマウントポイントを NFS や CIFS で共有するのが唯一の方法です。

### ブロックサイズはどれくらいが良いか?

mkfs.s3ql の `--data-block-size` オプションでブロック最大サイズを決めます。

ブロックサイズより小さいファイルには影響しません。大きいファイルは複数ブロックに分割され、アップロード/ダウンロードはブロック単位で行われます。例えば 10 MB ブロックのときに 100 MB ファイルの 5 バイトを書き換えると、約 10 MB 分の転送が発生します(圧縮で減る場合あり)。5 MB ブロックなら約 5 MB になります。

逆に 100 MB 全部を読むとき、10 MB ブロックなら 10 回のリクエスト、1 MB ブロックなら 100 回になります。リクエスト数が増えるほど固定コストが増え効率が悪くなり、また多くのプロバイダはリクエスト単位で課金するため 1×100 MB のほうが 100×1 MB より安くなります。

この 2 つのトレードオフでバランスを取ります。大きくすれば無駄な転送が増え、小さくすればリクエスト数が増えます。

よほど理由がなければデフォルトを使うのが良いでしょう。なおブロックサイズより小さいファイルには影響しないため、ほとんどのファイルが 1 MB 未満なら、デフォルトの 10 MB を 1 MB にしてもほとんど変わりません。

### オフラインでアクセスする方法は?

ありません。ただしほとんどの用途ではこの機能は実際には不要です。クラウドに保存しつつオフラインでもアクセスしたいなら、ローカルディレクトリと S3QL マウントポイントを定期的に同期すれば十分です。例えば永続キャッシュ付きの S3QL マウントポイントを `/mnt/data` にしたいなら:

1. `/mnt/data` にローカルブロックデバイスをマウント
2. 小さなキャッシュサイズ(例: `--max-cache-entries 10`)で `/mnt/data_online` に S3QL をマウント
3. `/mnt/data_online` と `/mnt/data` を以下で同期
   - cron で `rsync -aHA --delete-during --partial` を定期実行するか
   - [watch.sh](https://github.com/drunomics/syncd/blob/master/watch.sh) や [lsyncd](https://code.google.com/p/lsyncd/) で inotify ベースで継続監視
4. これで `/mnt/data` を永続キャッシュ付き S3QL のように使えます

## 既知の問題

- S3QL は重複排除を SHA256 チェックサムのみで判断し、バイト単位の比較は行いません。2 つのブロックが異なる内容で同じチェックサムを持つ可能性がある(衝突)ため、その場合は片方のデータがもう片方のデータで上書きされる問題が起こります。任意 2 ブロックの衝突確率は約 1/10^77(= 1/2^256)。総 10^34 ブロックのファイルシステムでは約 1/10^9 に上がります。10^34 ブロック以上(ブロックサイズ極小 4 KB でも約 10^25 TB)は非推奨です。極端な不運も問題になり得ます。
- ACL(Access Control Lists)には未対応です。原理的な障害はなく、実装する人を待っているだけです。
- Linux 3.5 時点で、ディレクトリの「書き込み禁止ビット」は未実装です。ビットが立っていてもディレクトリ所有者はその中のファイルや空サブディレクトリを削除できてしまいます。これは FUSE カーネルモジュールのバグ([issue #23](https://github.com/libfuse/libfuse/issues/23))で、カーネル側で修正が必要です。2016/2/28 時点で近い修正予定はなさそうです。
- S3QL は `relatime` マウントオプション相当の動作を常に行います。atime は現在値が ctime または mtime より古い場合のみ更新されます。
- S3QL のディレクトリは `st_nlink` が常に 1 です。`st_nlink = 2 + サブディレクトリ数` を前提とするプログラムは混乱する可能性があります。これは S3QL のバグではなく、サブディレクトリ数を nlink に含めるのは UNIX の慣例に過ぎず要件ではありません。この慣例に盲目的に依存するアプリがあれば、それはそのアプリのバグです。有名な例は古い GNU find で、S3QL では `--noleaf` オプションが必要でしたが、最近の find では修正されています。
- `umount` や `fusermount -u` は全データのアップロード完了を待ちません(FUSE 制約 [issue #1](https://github.com/libfuse/libfuse/issues/1))。これらを使う場合は、シャットダウン前に `mount.s3ql` プロセスの終了を明示的に待つ必要があります。したがって `/etc/fstab` でのマウントは非推奨(専用 init スクリプト推奨)。
- S3QL はバックエンドが容量不足にならないことを前提としています。大手(Amazon S3、Google Storage)なら保証されますが、自前サーバや小規模プロバイダでは問題になり得ます。容量不足に陥るとファイルシステムは不整合状態になり、fsck が必要になります(fsck 前にバックエンドの容量を確保してください)。残念ながら容量不足を不整合なく扱う方法はなく、S3QL は書き込みをキャッシュに入れた後で「バックエンドに送れない」と判明したときエラーを返せないためです。

## 実装の詳細

この節は S3QL の内部動作の背景情報です。利用するだけなら読む必要はありません。

### メタデータストレージ

S3QL は多くの UNIX ファイルシステム同様、アイノード(inode)概念を持ちます。

ディレクトリアイノードの中身(ディレクトリ内のファイル・サブディレクトリ名と inode)は [SQLite](http://www.sqlite.org/) データベースに直接格納されます。このデータベースは複数のストレージオブジェクトに分割され、マウント時にダウンロードされ、バックグラウンドで定期的に、およびアンマウント時にアップロードされます。2 つの意味合いがあります:

1. ファイルシステムツリー全体をデータベースから読み出せます。バックエンドとのやり取りはファイル(正確には inode)の中身アクセスにのみ必要で、多くの操作がネットワーク通信なしで済むため高速です。
2. S3QL ファイルシステムは一度に 1 台でしかマウントできません。さもないと、一方の変更が他方のアンマウント時に上書きされます。

ソケット、FIFO、キャラクタデバイスには追加のストレージは不要で、情報はすべてデータベース内にあります。

### データストレージ

ファイル inode の中身は個別のブロックに分割されます。最大ブロックサイズはファイルシステム作成時に決定され、後から変更できません。各ブロックはバックエンドに 1 オブジェクトとして保存され、inode → ブロック、ブロック → オブジェクトのマッピングはデータベースに保持されます。

マウント中はブロックがローカルキャッシュされます。

ブロックはバックエンド保存前に圧縮・暗号化が可能です(アップロード時に行われる。キャッシュ内は非圧縮・非暗号化)。

複数ファイルが同一内容のブロックを持つ場合、同じバックエンドオブジェクトに保存されます(データは 1 度のみ保存)。

### データ重複排除

各ブロックをアップロードする前に SHA256 ハッシュを計算し、同じハッシュのブロックが既存なら新規アップロードを行わず、既存オブジェクトへのリンクにします。

ユーザーからは透過的で、ブロック内容は後から変更可能です。共有中のブロックの片方が変更された場合は、変更されたブロックを新しいオブジェクトに自動的に保存し、他方のブロック内容は不変のままとなります。

### キャッシュ

アプリがファイルを読み書きするとき、S3QL は必要なブロックを特定し、バックエンドから取得(または新規作成)します。そのブロックをキャッシュディレクトリに保持し、数秒間アクセスがなければ S3 へ書き込みます。ブロックがキャッシュから削除されるのは最大キャッシュサイズに達したときだけです。

アンマウント時には変更ブロックを全部バックエンドに書き出し、キャッシュをクリーニングします。

### 暗号化

ファイルシステム作成時に **mkfs.s3ql** が `/dev/random` から 256 ビットのマスターキーを生成します。マスターキーはユーザー入力パスフレーズで暗号化され、他のファイルシステムデータと共に保存されます。パスフレーズはマスターキーへのアクセスにしか使わないため、容易に変更可能です。

データはオブジェクト・アップロードごとに新しいセッションキーで暗号化されます。セッションキーはマスターキーに nonce を連結して SHA256 したものです。nonce はオブジェクト ID と現在の UTC 時刻(32 ビット浮動小数)を連結したもので、精度は Python の [time()](http://docs.python.org/library/time.html#time.time) 関数に依存(通常 1 ms 以上)。SHA256 実装は Python 標準ライブラリを使用。

セッションキー計算後、アップロード対象データに対し SHA256 HMAC を計算します。その後データを圧縮し(mount.s3ql に `--compress none` を指定した場合を除く)、先頭に HMAC を挿入。HMAC + 圧縮データを [PyCrypto](http://www.pycrypto.org/) の AES-256 CTR モードで暗号化します。最後に nonce を先頭に付加し、新しい S3 オブジェクトとしてバックエンドに送信します。
